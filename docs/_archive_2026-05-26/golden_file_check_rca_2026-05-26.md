# Golden File Check — Root Cause Analysis (2026-05-26)

## TL;DR

The Golden File Check workflow is **NOT broken in main**. The failing PRs (#20, #24, #36, #38) all have **stale GitHub-generated merge refs** that were created before PR #39 (`1658698 fix(ci): gate Golden File Check require-step on real detector output`) landed in main. Those stale merge trees still contain the OLD, buggy `golden_check.yml` whose detector always reported `modifies_goldens=true`.

**Fix per affected PR: re-trigger the merge ref** by pushing a trivial commit, closing+reopening the PR, or rebasing onto current main. No code/workflow change needed in main.

## Workflow Definition Summary

`.github/workflows/golden_check.yml` (current, post-PR-#39):

```yaml
name: Golden File Check
on:
  pull_request:
    branches: [main]
jobs:
  check:
    runs-on: ubuntu-22.04
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Check if PR modifies golden tests
        id: check_goldens
        run: |
          changed_files=$(git diff origin/main...HEAD --name-only)
          if echo "$changed_files" | grep -qE "test_.*golden|golden.*test|snapshot.*test|expected_.*\.json"; then
            echo "modifies_goldens=true" >> $GITHUB_OUTPUT
            ...
          else
            echo "modifies_goldens=false" >> $GITHUB_OUTPUT
            ...
          fi

      - name: Require [regen-goldens] tag in commit message
        if: steps.check_goldens.outputs.modifies_goldens == 'true'
        run: |
          if git log origin/main..HEAD --pretty=format:"%s%n%b" | grep -qE "\[regen-goldens?\]"; then
            echo "[OK] [regen-goldens] tag found"
          else
            echo "[FAIL] Golden files modified but no [regen-goldens] tag in any commit message"
            exit 1
          fi
```

**Intent:** detect PRs that modify golden snapshot files; require `[regen-goldens]` tag in a commit message before allowing the change.

**The bug PR #39 fixed:** the old detector piped `grep` output into `head -5`, masking grep's exit status. `head` always exits 0 even on empty input, so the `if` always succeeded → `modifies_goldens=true` on every PR → require-step always ran → typical PRs failed because they have no `[regen-goldens]` tag.

## Root Cause

GitHub's `refs/pull/N/merge` ref is a **synthetic merge commit** generated by GitHub between the PR HEAD and the base branch at a moment in time. It is NOT automatically regenerated when main moves forward; it is only regenerated on:

1. New commit pushed to the PR HEAD branch
2. PR closed and reopened
3. The "Update branch" button on the PR is clicked (creates a merge commit on the PR HEAD)
4. PR is rebased

For PRs #20, #24, #36, #38, the merge refs were created or last refreshed BEFORE PR #39 (`1658698`) landed in main on 2026-05-25 night. Their merge trees still embed the pre-fix `golden_check.yml`. When the workflow re-runs (e.g. on a push to main triggering rebase-tests-of-open-PRs, or manual rerun), GitHub Actions checks out the stale merge ref and loads the OLD buggy YAML.

PR #39 landed at commit `1658698`, between `8abe675` (in failing PRs) and `8073bcc` (in passing PRs).

Critically: **none of the four affected PRs touch golden files** (verified via `gh pr view N --json files`). So once their merge refs are regenerated against current main, the (now-fixed) detector will correctly report `modifies_goldens=false` and the check will pass.

## Fix Recommendation

### NO workflow change needed.

The workflow `.github/workflows/golden_check.yml` already has the fix from PR #39 in main. No further changes to the workflow are warranted — adding more defensive logic would mask the underlying root cause (stale merge refs), not fix it.

### Per-PR action plan (rebase / refresh)

For each of PR #20, #24, #36, #38: the maintainer (or a delegated agent) needs to regenerate the merge ref. Options in priority order:

1. **Rebase onto current main** (preferred — also brings the PR up to date):
   ```bash
   git fetch origin
   git checkout pr-64-cross-platform-ci-matrix     # PR #20
   git rebase origin/main
   git push --force-with-lease origin pr-64-cross-platform-ci-matrix
   # repeat for pr-69-docs-refresh-v1.7.1-v1.8 (PR #24),
   #            pr-77-range-fractional-spec (PR #36),
   #            pr-76-exploitative-play (PR #38)
   ```

2. **Empty commit nudge** (lighter — only refreshes merge ref):
   ```bash
   git checkout <pr-branch>
   git commit --allow-empty -m "chore: trigger merge ref refresh post PR #39"
   git push origin <pr-branch>
   ```

3. **Close + reopen PR** via UI or `gh pr close N && gh pr reopen N` (also forces ref refresh).

After any of these, the Golden File Check should pass for the four affected PRs without further intervention (since none touches golden files).

### Stale branch cleanup

`origin/pr-80-golden-file-check-gating-fix` contains the same fix as PR #39 (commit `8b6f142` is the standalone version of `1658698`). It is now redundant. Recommend deleting this branch:

```bash
git push origin --delete pr-80-golden-file-check-gating-fix
```

(Only if no open PR uses it — verified above, no PR #80 exists. The branch was pushed before PR #39's commit was retitled/merged.)
