# v0.3.1 patch release notes

**Released:** 2026-05-21 (within the same autonomous session as v0.3.0)

## What changed

Two bug fixes caught during the PR 3.5 ready-to-commit verification chain.
Both are correctness-related, not feature additions. No new public API; no
config / schema changes; no migration required.

### Fix 1: `get_full_range` returns all 169 canonical hand classes

The DCFR chart generator (`scripts/generate_pushfold_charts.py`) writes
chart cells in sparse form — any (hand_class, frequency) pair where
`frequency == 0.0` is omitted from the JSON to keep `pushfold_v1.json`
under control. At 10 BB SB jam, only 113 of the 169 canonical classes
have non-zero jam frequency; the other 56 trash hands are not stored.

The single-hand path (`get_pushfold_strategy`) handled this correctly via
a `chart.get(canonical, 0.0)` default in `poker_solver/pushfold.py:124`,
so end users querying e.g. `AKs` always got the right number. But
`get_full_range` previously returned the chart cell as-is, so a caller
asking for the full grid at 10 BB SB jam got a dict with 113 keys
instead of 169. Any downstream consumer that iterated over the dict
or zipped it against a fixed 169-class basis (e.g. for visualization
or matrix construction) would silently drop the trash region.

**Fix:** `poker_solver/pushfold.py:128-137` now explicitly fills the
returned dict with every canonical class, defaulting absent hands to
`0.0`:

```python
def get_full_range(stack_bb: int, position: str) -> dict[str, float]:
    pos = _validate_stack_and_position(stack_bb, position)
    chart = _get_chart_cell(stack_bb, pos)
    return {cls: float(chart.get(cls, 0.0)) for cls in _all_hand_classes()}
```

Helper `_all_hand_classes()` (`poker_solver/pushfold.py:140-153`)
enumerates the 13 pairs + 78 suited + 78 offsuit canonical ordering
deterministically.

**Regression test:** existing PR 3.5 tests in `tests/test_pushfold.py`
already asserted the full-range shape; the discrepancy surfaced when
the chart file was regenerated by Agent B's DCFR run and the test
counts started failing. See autonomous_log.md S9 for the discovery
trail.

### Fix 2: Push/fold dispatch requires `starting_street == PREFLOP`

The original PR 3.5 dispatch condition in `poker_solver/solver.py`
checked only "is this a `HUNLPoker` with effective stack in [2, 15] BB."
That works for the canonical case (someone calls `solve()` on a fresh
preflop game at short stack), but it silently misfires on a HUNL
**subgame** whose starting street is not preflop. The most concrete
example: `default_tiny_subgame()` is a river-only AhKc-vs-QdQh fixture
with `starting_stack = 1000 cents = 10 BB`. Under the old dispatch,
`solve(HUNLPoker(default_tiny_subgame()))` would route to the push/fold
chart lookup — which returns a preflop jam/fold equilibrium that has
zero relationship to the river spot the caller actually wants solved.

Push/fold equilibria only exist on the **preflop tree shape**; a 10 BB
river subgame is still a real river spot that must run through the
tree solver. The dispatch needed a `starting_street` guard.

**Fix:** `poker_solver/solver.py:48-56` now also requires preflop start:

```python
# Push/fold short-stack fast path: only dispatch for PREFLOP-start games.
# A river/turn/flop subgame at short stack still needs the tree solver —
# push/fold equilibria only exist on the preflop tree shape.
if (
    isinstance(game, HUNLPoker)
    and game.config.starting_street == Street.PREFLOP
    and is_pushfold_mode(game.config.starting_stack, game.config.big_blind)
):
    return _solve_pushfold_lookup(game.config)
```

**Regression test:** `tests/test_pushfold.py:144-161`,
`test_pushfold_mode_not_triggered_for_river_subgame_at_short_stack`.
Asserts that `solve(HUNLPoker(default_tiny_subgame()))` does NOT return
`backend == "pushfold"`, validating that the river-subgame fixture
runs through the DCFR path (or, at worst, raises) rather than silently
returning a preflop chart answer.

This bug was caught by the architecture-doc agent — a parallel agent
writing `docs/architecture.md` that walked through the dispatch logic
and realized the guard was insufficient. The agent reported the
discrepancy, the fix and regression test were added immediately, and
the chain re-verified before any PR 3.5 code shipped.

## Why this matters

Both bugs share a pattern: they don't crash, they don't fail loudly,
and they don't show up in the "happy path" tests that PR 3.5 ships
with. They would have manifested as **silently-wrong answers** for any
caller who:

- iterated `get_full_range(...)` against a fixed 169-class basis (e.g.
  to build a visualization, a heatmap, or feed it into a downstream
  matrix solver — anything that doesn't tolerate dropped keys); or
- called `solve()` on a HUNL subgame whose starting street is not
  preflop and whose stack depth happens to be in [2, 15] BB — the
  exact shape of the `default_tiny_subgame()` fixture, which is
  used in `tests/test_hunl_tree.py` as the end-to-end smoke test.

If `default_tiny_subgame()` had been the next thing a downstream user
ran after the v0.3.0 release, they would have gotten back a SolveResult
whose `backend == "pushfold"` and whose `average_strategy` was the
10 BB SB jam chart — confidently, instantly, and silently wrong for
a river spot. That kind of failure is much worse than an exception:
it propagates into any analysis built on top of the result.

Neither bug ever shipped publicly — v0.3.0 was tagged but never
distributed before these fixes landed. Catching them inside the
autonomous session preserves the invariant that the public v0.3 line
is correctness-clean.

## Migration

None required. v0.3.0 was never distributed before v0.3.1, so v0.3.1
is the effective first public v0.3. No API surface changes; no JSON
schema changes; existing callers see strictly correct behavior where
the v0.3.0 version would have silently misbehaved on the two paths
above.

## Operational note

These bugs were caught by the project's standing **parallel-agents +
cross-check** discipline (see `PLAN.md` §5 and the agent-scheduling
rules in user memory): when PR 3.5 implementation went out to one
agent fan-out, an unrelated agent was simultaneously writing
`docs/architecture.md` against the same code. That agent's job was
not to find bugs — it was to document dispatch flow. But documenting
dispatch flow required reading the dispatch condition carefully, which
surfaced Fix 2. The sparse-JSON discrepancy (Fix 1) surfaced the same
way: the chart generation agent (Agent B in the PR 3.5 fan-out)
landed a regenerated chart file whose key count didn't match what
the lookup agent (Agent A) had assumed.

Sequential single-agent execution would likely have shipped both bugs.
The cross-check came for free from running the work in parallel and
having each agent verify against the same source of truth. This is
the second time in two PRs the pattern has paid off (PR 3 caught the
ALL-IN-CALL street-completion bug the same way — Agent A's tests
flagged a state-machine bug that Agent C's tests would not have
exercised).
