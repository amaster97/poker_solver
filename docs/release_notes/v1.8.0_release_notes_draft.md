## Headline

v1.8.0 makes the Discounted CFR solver fast on **every machine you run
it on** — not just Apple Silicon. The hot inner loops are now
hand-vectorized across three instruction-set families, with an automatic
scalar fallback for anything older or unrecognized.

If you're on:

- a recent **Apple Silicon Mac** (M1, M2, M3, M4): the NEON path lights
  up.
- an **Intel or AMD desktop / laptop / Intel Mac** with a CPU from
  ~2013 or later (Haswell+): the AVX2 path lights up.
- an **older x86_64 machine** without AVX2: the SSE2 path lights up
  (still faster than scalar).
- **anything else** (Linux on ARM servers, exotic arches, very old
  CPUs): the scalar path runs unchanged.

You don't have to configure anything. The library detects what your CPU
supports at runtime and picks the fastest implementation it can. All
paths produce **bit-identical** output — verified by an end-to-end
cross-backend smoke test that runs 1000 iterations of a Kuhn poker solve
on both the scalar and SIMD paths and asserts equality.

## What's new

### Cross-platform SIMD vector kernels (PRs #23, #25, #26, #32, #70)

Four hot loops in `dcfr_vector.rs` are now vectorized:

| Kernel                  | Phase | PR        |
|-------------------------|-------|-----------|
| `discount_regrets`      | 1     | #23 (NEON + SSE2), #25 (AVX2) |
| `discount_strategy_sum` | 1     | #23, #25  |
| `update_regret_sum`     | 2     | #26       |
| `update_strategy_sum`   | 3     | #70       |
| `compute_strategy`      | 4     | #32       |

Each kernel ships three implementations behind a single safe public API:

1. **NEON** for `target_arch = "aarch64"` (Apple Silicon, ARM servers).
2. **AVX2 + SSE2** for `target_arch = "x86_64"`, with an
   `is_x86_feature_detected!("avx2")` runtime check that falls back to
   SSE2 on pre-Haswell CPUs.
3. **Scalar** for any other architecture, and for the epilogue (the 0-3
   trailing elements that don't fit a SIMD lane).

### Why this matters

The previous v1.7.x line was already SIMD-optimized for Apple Silicon
(via NEON), but x86_64 users (Intel Macs, Windows PCs, Linux desktops,
CI runners) got the scalar fallback. That's a 4-8x speedup gap for
roughly half the user base. v1.8.0 closes that gap.

### Performance expectations

Per the v1.8 SIMD spec (`docs/v1_8_cross_platform_simd_spec.md`),
measured against the v1.7.0 scalar baseline on the per-iter hot loops:

- **Apple Silicon (M-series)**: ~4-8x faster
- **x86_64 with AVX2** (~2013+ Intel/AMD): ~2-4x faster
- **x86_64 SSE2-only** (pre-Haswell): ~1.5-2x faster
- **Non-vectorized arches**: no regression (scalar path unchanged)

End-to-end wall-clock improvement depends on workload (how much time is
spent in vectorized inner loops vs. tree traversal, range expansion,
etc.) — but the per-iter speedup unblocks the Sarah persona W2.3
benchmark (turn Nash under a 5-min budget) on M-series.

### CI hardening (PRs #20, #21, #22, #28)

- **PR #20**: cross-platform CI matrix (Ubuntu x86_64 + macOS arm64) so
  every PR is built and tested on both architectures.
- **PR #21**: CI-driven release workflow.
- **PR #22**: Ship-hardening Guards B + C (sanity gates on release).
- **PR #28**: Brown reference build script self-bootstrap (CI no longer
  hits an unbootstrapped-toolchain bug).

## Compatibility

- **No public API changes.** All vectorization is behind the existing
  Rust extension's safe Python API.
- **No CFR algorithm changes.** Bit-identical output to v1.7.0 on the
  same inputs.
- **Min Python**: unchanged from v1.7.0.
- **Min Rust toolchain**: unchanged from v1.7.0.

If you observe non-bit-identical output between v1.7.0 and v1.8.0 on
the same problem, please open an issue — this is a regression by
definition.

## Verification

Run the cross-backend smoke test yourself:

```bash
cargo test --release --features simd -- --nocapture cross_backend_smoke
```

This solves Kuhn poker for 1000 DCFR iterations on both the scalar and
SIMD paths and asserts bit-identical regret + strategy_sum tables at
every iteration.
