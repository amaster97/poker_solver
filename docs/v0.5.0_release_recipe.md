## 2. Tag recipe for v0.5.0

### Decision point: tag on `integration` or wait for `main` merge?

**Recommended: tag on `main` after the main merge.** Reasons:

1. GitHub's default release-branch dropdown points at `main`; tagging there
   makes the release node land where users expect.
2. `main` is the long-lived public branch on `origin`; `integration` is
   an internal staging branch and may be rewound or pruned later.
3. Per-PR-branch policy (memory) treats `integration` as the merge target,
   not the release target. `main` is the release branch.
4. Future tags (v0.6.0 from PR 7, v1.0 etc.) should follow the same
   convention. Setting it now avoids a v0.5.0 outlier.

**Counter-argument (rejected):** tagging on `integration` immediately would
ship the release notes faster, but the few hours saved aren't worth the
release-branch inconsistency.

### Commands (run **only after** user approves main merge)

```bash
# 1. Confirm main is up to date with the PR-6 merge:
git checkout main
git pull origin main
git log --oneline -3
# Expect to see the PR-6 merge commit at HEAD.

# 2. Create the annotated tag at main HEAD:
git tag -a v0.5.0 -m "v0.5.0: Rust HUNL postflop port (~24x speedup)"

# 3. Verify:
git tag -l "v*"
git show v0.5.0 --stat | head -20

# 4. Push the tag to origin:
git push origin v0.5.0
```

**If the user prefers to tag a specific SHA** (e.g. the integration tip
before the main merge), substitute step 2:

```bash
git tag -a v0.5.0 6c438b8 -m "v0.5.0: Rust HUNL postflop port (~24x speedup)"
```

## 3. GitHub release recipe

### Option A — one-shot CLI (preferred, faster)

```bash
gh release create v0.5.0 \
  --notes-file docs/release_notes_v0.5.0.md \
  --title "v0.5.0: Rust HUNL postflop port (~24x speedup)"
```

This creates the release **and** publishes it. The tag must already exist on
origin (see section 2 step 4).

### Option B — draft first, publish after review

```bash
gh release create v0.5.0 \
  --notes-file docs/release_notes_v0.5.0.md \
  --title "v0.5.0: Rust HUNL postflop port (~24x speedup)" \
  --draft
```

Then review at `https://github.com/amaster97/poker_solver/releases` and click
**Publish release** in the UI when ready.

### Option C — pure UI

1. Visit `https://github.com/amaster97/poker_solver/releases/new`
2. Choose tag `v0.5.0` (must exist on origin)
3. Title: `v0.5.0: Rust HUNL postflop port (~24x speedup)`
4. Paste contents of `docs/release_notes_v0.5.0.md` into the body
5. Click **Publish release** (or **Save draft**)

## 4. Recommended sequence

1. **User approves main merge of `integration`** (separate decision, not
   covered by this recipe).
2. **User merges `integration` into `main`** locally and pushes — or merges
   via GitHub PR UI.
3. **User runs section 2 commands** (`git tag -a v0.5.0 ...` + push).
4. **User runs section 3 Option A or B** (`gh release create ...`).
5. **Verify** at `https://github.com/amaster97/poker_solver/releases/tag/v0.5.0`.

Steps 3–4 can be a single shell session; ~30 seconds of work.

---

## 5. Constraints / do-not-do

- **Do not tag autonomously.** This recipe stops at "pre-staged"; the user
  must explicitly authorize each step.
- **Do not push `--force` to `main`.** `git push origin v0.5.0` is a pure
  additive tag push; no force needed.
- **Do not amend the release notes** at
  `docs/release_notes_v0.5.0.md` between drafting and publishing without
  the user's review — that file is the source of truth for the published
  release body.
- **Do not delete `integration`** after the main merge; it stays as the
  staging branch for PR 7+.
- **No-skip-hooks rule applies** (per memory): the tag push and release
  creation do not invoke hooks, but any follow-on commits must.

## 6. Rollback (if needed)

If the release goes out and a problem is discovered:

```bash
# Delete the GitHub release (keeps the tag):
gh release delete v0.5.0

# Delete the local tag:
git tag -d v0.5.0

# Delete the remote tag:
git push origin :refs/tags/v0.5.0
```

Then re-cut once the fix lands. Tags are cheap; don't be precious about
pulling a bad one.
