## 2. Steps to release v1.0.0 to main (awaits user OK)

### 2.1 Pre-flight checks

```bash
# Confirm working tree is clean and on integration
git status
# Expect: "nothing to commit, working tree clean"

# Confirm integration is in sync with origin
git fetch origin
git log --oneline origin/integration..integration
git log --oneline integration..origin/integration
# Both should be empty (integration == origin/integration).

# Confirm v1.0.0 tag points where we expect
git rev-parse v1.0.0
# Expect: bbb4395... (the integration tip commit)

# Confirm main is at 2b67370 and untouched
git rev-parse origin/main
# Expect: 2b67370...
```

If any check fails, **stop** and resolve before proceeding.

### 2.2 Merge integration into main

```bash
# 1. Switch to main and pull
git checkout main
git pull origin main
# main should be at 2b67370

# 2. Merge integration with a no-ff merge commit
git merge --no-ff integration -m "v1.0.0 GA release"

# 3. Push the merge to origin
git push origin main
```

The `--no-ff` flag preserves the integration branch's history as a
visible merge commit, which matches the v0.5.0 / v0.6.0 convention and
makes the GA cut point easy to find with `git log --first-parent main`.

### 2.3 Optional: move v1.0.0 tag to main merge commit

If you prefer the tag on the main branch (consistent with v0.5.0 policy):

```bash
# After step 2.2 completes, capture the new main HEAD
NEW_MAIN_SHA=$(git rev-parse main)

# Delete the local tag and re-create it on the main merge commit
git tag -d v1.0.0
git tag -a v1.0.0 "$NEW_MAIN_SHA" -m "v1.0.0: GA release (post-v1 hardening)"

# Force-update the remote tag (this is a tag move, not a branch force-push)
git push origin :refs/tags/v1.0.0      # delete remote tag
git push origin v1.0.0                  # push new tag
```

**If you keep the tag on `integration`** (current state, `bbb4395`):
no action needed. The release notes still apply; GitHub releases can
reference a tag on any branch.

### 2.4 Optional: create GitHub release

```bash
gh release create v1.0.0 \
  --notes-file docs/release_notes_v1.0.0.md \
  --title "v1.0.0: GA release"
```

Or draft-first:

```bash
gh release create v1.0.0 \
  --notes-file docs/release_notes_v1.0.0.md \
  --title "v1.0.0: GA release" \
  --draft
```

Then review at `https://github.com/amaster97/poker_solver/releases` and
publish in the UI.

## 3. Rollback plan

### 3.1 If the main merge breaks something before push

```bash
# Still local, not yet pushed — reset to the prior main SHA
git reset --hard 2b67370
```

### 3.2 If the merge has been pushed and a problem is found

```bash
# Capture the pre-merge SHA before doing anything else
PREV_MAIN_SHA=2b67370

# Reset local main and force-push (destructive — coordinate with collaborators)
git checkout main
git reset --hard "$PREV_MAIN_SHA"
git push --force-with-lease origin main
```

**Warning:** force-pushing `main` is destructive. Only do this if the
merge has not been pulled by anyone else and the user explicitly
authorizes. Memory rule: **never force-push main without explicit user
approval.** Prefer a revert commit (`git revert -m 1 <merge-sha>`) if
the merge has propagated.

### 3.3 If the GitHub release is bad but the merge is fine

```bash
gh release delete v1.0.0   # keeps the tag
# Edit docs/release_notes_v1.0.0.md, then re-create:
gh release create v1.0.0 --notes-file docs/release_notes_v1.0.0.md --title "..."
```

### 3.4 If the v1.0.0 tag itself needs to move

```bash
git tag -d v1.0.0
git push origin :refs/tags/v1.0.0
git tag -a v1.0.0 <new-sha> -m "v1.0.0: GA release"
git push origin v1.0.0
```

Tags are cheap; don't be precious about pulling a bad one.

## 6. Constraints / do-not-do

- **Do not run autonomously.** This recipe stops at "RECIPE"; the user
  must explicitly authorize each step.
- **Do not force-push `main`** unless rollback is explicitly authorized
  and no collaborator has pulled the bad merge.
- **Do not skip hooks** on any follow-on commits (per memory).
- **Do not delete `integration`** after the merge; it stays as the
  staging branch for post-GA work.
- **Do not amend `docs/release_notes_v1.0.0.md`** between drafting and
  publishing without user review.
- **Do not merge `integration` into `main` via `--squash`** — that would
  lose the per-PR commit history that the integration branch preserves.
